## Season Calendar

| Event               | T1 (NBA)                       | T2 (NARBL)                     | T3 (MBL)                       |
|---------------------|--------------------------------|--------------------------------|---------------------------------|
| Season start        | 3rd Tue of Oct (~Oct 15)       | 1st Tue of Nov (~Nov 1)        | 1st Tue of Dec (~Dec 1)         |
| Games per team      | 82                             | 60                             | 40                              |
| All-Star break      | Feb 13–18 (all tiers)          | Feb 13–18 (all tiers)          | Feb 13–18 (all tiers)           |
| Trade deadline       | Mar 5 (all tiers)              | Mar 5 (all tiers)              | Mar 5 (all tiers)               |
| Regular season end  | Apr 12 (all tiers)             | Apr 12 (all tiers)             | Apr 12 (all tiers)              |

All dates are generated by `CalendarEngine.getSeasonDates(startYear)`. Tiers start staggered but end simultaneously. All three tiers share the same All-Star break and trade deadline dates.


## Schedule Structure

Each tier has its own schedule array stored on gameState:
- `gameState.tier1Schedule` — 82 games per team, 30 teams
- `gameState.tier2Schedule` — 60 games per team, 86 teams across 11 divisions
- `gameState.tier3Schedule` — 40 games per team, 144 teams across 24 metro divisions

Schedules are generated by `CalendarEngine.generateCalendarSchedule()` during season setup (`OffseasonController.continueToSeasonSetup()`). Each schedule entry is a game object:

```
{ date: "YYYY-MM-DD", homeTeamId, awayTeamId, played: false,
  homeScore, awayScore, winnerId, boxScore }
```

T1 uses a structured NBA-style scheduling model (in-division, in-conference, cross-conference quotas — 16 + 36 + 30 = 82). T2/T3 use division-weighted random matchups via `_generateDivisionMatchups()`.

Box scores are only stored for the user's games to keep save sizes manageable.


## Simulation Pipeline

### Layer 1: Game Engine

`GameEngine` (332 lines) — possession-by-possession basketball simulation. Pure logic, no state management. Determines shot selection, makes/misses, rebounds, turnovers, fouls based on player ratings.

`GamePipeline` (989 lines) — wraps GameEngine into a complete game. `GamePipeline.create(homeTeam, awayTeam, options)` returns an object with `tick()` (advance one play), `getResult()` (final stats), and state accessors for live watch mode. Handles quarter breaks, overtime, fatigue accumulation, lineup management.

`SimulationController` (235 lines) — thin orchestration layer. `simulateFullGame()` runs a complete game via GamePipeline and accumulates stats. `simulatePlayoffGame()` does the same with playoff flags. `accumulatePlayerStats()` writes per-game stats to player objects and team totals.

### Layer 2: Day Simulation

`GMMode._simulateAllGamesOnDate(dateStr, silent)` — the workhorse. For a given date:
1. Gets games for all 3 tiers via `CalendarEngine.getGamesForDate()`
2. For each unplayed game: applies fatigue auto-rest, simulates via `SimulationController`, marks played, records result
3. For user games: stores detailed box score, checks for injuries, emits events
4. For all games: processes fatigue after game, updates injuries
5. If user team has injuries: stores them as `gameState.pendingInjuries` for modal display

### Layer 3: Sim Controls

`GMMode` (1,035 lines) — the regular season orchestrator. Receives sim commands from the UI and manages the day-by-day flow.

| Method | Triggered by | Behavior |
|--------|-------------|----------|
| `simulateNextGame()` | "Sim Game" button | Advances to user's next game date, sims all games on that date. If date is past All-Star and All-Star hasn't run, triggers All-Star first. |
| `simulateDay()` | "Sim Day" button | Sims all games on current date. Advances date by 1. Checks for calendar events, AI trades, injuries. |
| `simulateWeek()` | "Sim Week" button | Batch-sims 7 days. Interrupts for injuries, AI trade proposals, and breaking news. Resumes via `_resumeSimWeek()` callbacks. |
| `finishSeason()` | "Finish Season" button | Batch-sims all remaining games. Runs All-Star silently if needed. Interrupts for injuries and All-Star UI. Calls `showSeasonEnd()` when all tiers complete. |
| `finishSeasonBatch()` | Internal | The batch loop inside finishSeason. Processes days in a while loop, yields for injuries and All-Star. |

### Watch Game Flow

`GameSimController.watchNextGame()` — sets up a live play-by-play game:
1. Finds user's next game from schedule
2. Sims all OTHER games on that date (silent)
3. Creates a `GamePipeline` instance (`this._watchGame`)
4. Opens `WatchGameModal` via `window._reactShowWatchGame()`
5. Starts `_startWatchTimer()` — interval that calls `tick()` on the pipeline
6. `watchGameSetSpeed(speed)` adjusts the interval (1x, 2x, 4x, 8x)
7. On game complete or user close: `watchGameClose()` records result, accumulates stats, processes injuries/fatigue, saves, refreshes UI

The same watch infrastructure is reused for playoff games via `watchCalendarPlayoffGame()` on GameSimController.


## Mid-Season Events

### All-Star Weekend (Feb 13–18)

Triggered when sim advances past the All-Star start date. `GMMode.simulateNextGame()` detects when the next user game is past All-Star and:
1. Sims all remaining games up to All-Star date
2. Calls `deps.runAllStarWeekend()` which invokes `StatEngine.selectAllStars()` for each tier
3. Shows `AllStarModal` with selections and game results
4. After user dismisses, advances date past All-Star end and continues

In `finishSeason()`, All-Star is run silently (no modal) if not yet completed.

State flag: `gameState._allStarCompleted` — prevents re-trigger on subsequent sims.

### Trade System

`TradeEngine` (421 lines) — pure evaluation logic. Trade scoring, fairness checks, AI valuation.
`TradeController` (617 lines) — orchestrates user and AI trade flows.

**AI-to-AI trades**: `GMMode.processAiToAiTrades(date)` runs after each sim day. AI teams evaluate and execute trades based on needs and valuations. Returns notable trades for breaking news display.

**User trades**: Accessible via Trade screen in sidebar. User proposes trades evaluated by TradeEngine fairness logic. AI counter-offers possible.

**Trade deadline**: Mar 5 (all tiers). After this date, `processAiToAiTrades` stops executing and the trade UI is disabled.

### Injury System

`InjuryEngine` — generates injuries based on game minutes and fatigue. Three severity tiers with different recovery times.

Flow during sim:
1. After each user game, `checkForInjuries()` rolls for injuries on both teams
2. AI team injuries are auto-resolved (rest or play-through based on severity)
3. User team injuries stored as `gameState.pendingInjuries`
4. `InjuryModal` shows each injury with "Rest" / "Play Through" options
5. After all injuries handled, sim resumes via callback (`_resumeAfterInjuries`)

In batch sim (simWeek, finishSeason), injuries interrupt the batch. The resume callback re-enters the batch loop.

### Fatigue System

`FatigueEngine` — tracks cumulative fatigue per player based on minutes played.

- `applyAutoRest(team, isPlayoffs)` — before each game, auto-rests highest-fatigue players
- `processAfterGame(homeTeam, awayTeam, isPlayoffs)` — after each game, accumulates fatigue from minutes
- Fatigue penalizes player performance ratings during simulation
- All fatigue resets during offseason player development phase

### Chemistry System

`ChemistryEngine` — team chemistry affects simulation outcomes. Chemistry builds over games played together, degrades with trades. Not directly manipulable by the user during regular season.


## Season Completion Flow

`GameState.isSeasonComplete()` returns true when ALL three tier schedules have every game marked `played`. This is checked:
- After every sim action in GMMode (`simulateNextGame`, `simulateDay`, `simulateWeek`, `finishSeason`)
- When true, calls `deps.showSeasonEnd()` which routes to `GameSimController.showSeasonEnd()`

`showSeasonEnd()`:
1. Sets `gameState.offseasonPhase = 'season_ended'`
2. Determines user's playoff status per tier rules (seed, relegation, safe)
3. Records season in `gameState.seasonHistory`
4. Computes awards for all 3 tiers via `StatEngine.calculateAwards()`
5. Stores everything in `gameState._seasonEndData`
6. Shows `SeasonEndModal` via `window._reactShowSeasonEnd()`

From SeasonEndModal, the user clicks "Begin Playoffs" which triggers the playoff flow (see `docs/PLAYOFFS-ALIGNMENT.md`).

## Design Decisions Made

1. **All tiers sim on same calendar**: When the user sims a day, ALL tiers' games on that date are simulated, not just the user's tier. This keeps the league world consistent. The calendar advances linearly through the shared date space.

2. **Staggered tier starts, shared end date**: T1 starts mid-October (82 games), T2 starts early November (60 games), T3 starts early December (40 games). All end April 12. This creates different game densities per tier.

3. **User games get detailed treatment**: Box scores, win probability history, and injury checks are only computed for the user's games. AI-vs-AI games use lightweight simulation (no events, no detailed stats) for performance.

4. **Interrupt-resume pattern for batch sim**: `simulateWeek()` and `finishSeason()` use callbacks to yield control for injuries, AI trade proposals, and breaking news, then resume the batch. This keeps the UI responsive without web workers.

5. **Single `_simulateAllGamesOnDate` for all paths**: Every sim action (next game, day, week, finish) routes through the same method. This ensures consistent behavior: fatigue, injuries, stat accumulation, and events all happen identically regardless of sim speed.

6. **Watch game reuses GamePipeline**: The live play-by-play viewer creates a `GamePipeline` instance and calls `tick()` on a timer. The same pipeline class is used for instant simulation — the only difference is whether ticks happen on an interval or in a loop.

7. **AI trades process daily**: `processAiToAiTrades()` runs on every sim day. Notable trades trigger a "Breaking News" display that interrupts sim flow. This creates the feeling of a living league.

8. **Trade deadline is universal**: All tiers share the Mar 5 deadline. No tier-specific trade windows.

9. **All-Star is a hard interrupt**: The All-Star break blocks sim advancement. In interactive mode (simNextGame, simDay), it shows the modal. In batch mode (finishSeason), it runs silently but still processes selections and game results.

10. **Season completion requires all tiers**: `isSeasonComplete()` checks all three tier schedules. Even if the user's tier is done, the season doesn't end until the other tiers' games are also complete (they get simulated along the way via the shared calendar).
